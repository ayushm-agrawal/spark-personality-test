# SPARK

**Speculative Personality Assessment with Reactive Kernel**

An LLM-driven adaptive personality test that generates questions dynamically based on prior answers, speculatively pre-computes likely next branches, and aggregates responses into a personality profile mapped to archetypes.

Live at [personality.ception.one](https://personality.ception.one).

---

## What it is

SPARK is the backend for a personality test that does not use a fixed question bank. Every question is generated by an LLM, conditioned on the user's prior answers, the assessment mode, and the traits the system still needs to measure. Answers are aggregated through a quality-weighted scoring pipeline that is resistant to gaming, and the final profile is mapped to a set of archetypes.

The point of the project was not the personality test itself. It was to see how far inference-time engineering could push a real-world adaptive LLM loop.

---

## What is interesting under the hood

### 1. Speculative prefetching of branches

Every question-generation call to the LLM returns more than just the next question. It also returns a `predicted_next_answer` field: a probability distribution over the answer options the user is most likely to pick. The backend uses this distribution to speculatively generate the *following* question for the top two predicted branches in parallel, staggered at 0ms and 200ms, cached with a 5-minute TTL.

When the user actually answers, the system checks the cache. If their choice matches a prefetched branch, the next question is served from cache with near-zero latency. If not, it falls back to a fresh LLM call. In practice most users hit a prefetched branch, so the assessment flows at the speed of a static form while remaining fully dynamic.

The same pattern (use the current call to predict and pre-compute the next one) shows up in agent inference infrastructure at frontier labs. Here it is applied to a consumer-facing adaptive test.

### 2. LLM output validation and auto-repair

Generated questions are not trusted by default. Each one is validated against a strict structural contract: the right trait is being measured, all required fields are present, and the score distribution across the answer options is exactly `{0, 1, 2}` (weak, medium, strong signal).

When the model returns a malformed distribution (for example `{2, 2, 1}`), the system does not throw. A `fix_question_scores` step sorts the options by the model's intended ordering and remaps them to the required distribution. Graceful degradation instead of user-visible failure.

### 3. Quality weighting under adversarial input

Not every submission is honest. The scoring layer detects gaming patterns in a user's response sequence: uniform answers, a/b/c repeating cycles, sub-two-second response times, answer sets clustered entirely in the middle of the score range. Detected tests are down-weighted or excluded from the holistic profile aggregation.

The shape of this mechanism, a domain-specific rule layer that handles edge cases a naive system would average away, is the same pattern that shows up in enterprise agent systems that need to survive contact with real users.

## Stack

- **FastAPI** for the HTTP layer
- **Azure OpenAI** for generation and branch prediction
- **Firestore** for session state and per-user assessment history
- **Jinja2** for modular, mode-aware prompt composition
- Deployed on Azure

The frontend is a separate React + Vite application (under `frontend-react/`) served at [personality.ception.one](https://personality.ception.one).

---

## Architecture

Request flow for a single assessment session:

1. **Session start.** The client requests a new session. The backend initializes state: the chosen assessment mode, the set of traits the test needs to measure, and an empty answer history.

2. **First question generation.** The backend composes a prompt via Jinja2 templates conditioned on mode and trait targets, calls Azure OpenAI, validates the response, and returns the first question. The response also includes `predicted_next_answer` probabilities.

3. **Speculative prefetch fires.** Before the user has seen the question, the backend spawns two parallel background tasks to generate the following question for the top two predicted answer branches. Results are cached.

4. **User answers.** The backend looks up the answer in the cache. If it matches a prefetched branch, the next question is served instantly. If not, a fresh LLM call is made. Either way, the cycle repeats from step 3.

5. **Session end.** When trait coverage targets are met, the answer history is passed through the quality-weighting layer (gaming detection plus weighted aggregation) and the result is mapped to a personality archetype profile.

## Local development

### Prerequisites

- Python 3.11+
- Node.js 20+ and npm 9+ (for the frontend)
- An Azure OpenAI deployment
- A Firebase project with Firestore enabled and a service account JSON

### Backend

1. Create a virtualenv and install dependencies:

   ```bash
   python -m venv .venv
   source .venv/bin/activate
   pip install -r requirements.txt
   ```

2. Copy `backend/.env.example` to `backend/app/.env` and fill in your values:

   ```bash
   cp backend/.env.example backend/app/.env
   ```

   Required variables:

   | Variable | Description |
   |---|---|
   | `AZURE_OPENAI_ENDPOINT` | Your Azure OpenAI resource endpoint |
   | `AZURE_OPENAI_API_KEY` | Key1 or Key2 from your Azure OpenAI resource |
   | `AZURE_OPENAI_MODEL_NAME` | Deployment name (e.g. `gpt-5.2-chat`) |
   | `GOOGLE_APPLICATION_CREDENTIALS` | Path to your Firebase service account JSON |

3. Place your Firebase service account JSON at `backend/app/serviceAccountKey.json` (gitignored).

4. Run the API:

   ```bash
   cd backend/app
   uvicorn main:app --reload --port 8000
   ```

### Frontend

See [frontend-react/README.md](frontend-react/README.md) for setup. In short:

```bash
cd frontend-react
npm install
npm run dev
```

The Vite dev server proxies `/api` to `http://localhost:8000`.
